# お見積書PDF リニューアル 検証ドキュメント

> 作成: 2026-05-26 / 対象: 見積もりメール添付のお見積書PDF

## 検証目的

メール添付の「御見積書」PDFを、添付された伝統的な日本式見積書テンプレートの**体裁（見た目）に寄せて見やすくする**。中身（従来型／ハイブリッドの2価格モデル・税別概算）は維持し、従来型表記は混同を招かないよう控えめにする。

## 対象範囲

| ファイル | 変更内容 |
|---|---|
| `estimate/components/pdf/EstimatePdf.tsx` | レイアウト骨格をテンプレ風に再構成 |
| `estimate/components/pdf/shared/PdfStyles.ts` | バナー・表・備考等のスタイル刷新（ブランドカラー維持） |
| `estimate/components/pdf/shared/PdfHeader.tsx` | 「御見積書」左上大型タイトル＋宛名＋項目欄＋会社情報の再構成 |
| `estimate/components/pdf/shared/PdfFonts.ts` | `NotoSansJP-Bold`（サブセット）を追加登録 |
| `estimate/public/fonts/` | Bold サブセットフォント追加 |
| `estimate/app/api/pdf/route.ts` | 必要に応じプロップ調整（精度・参考値の受け渡し） |

※ バックエンド (`backend/`) は PDF をフロントAPIに委譲しているため変更なしの見込み。

## リスク・確認観点（Codexレビュー用）
- フォントサブセットに静的ラベルの全文字が含まれているか（豆腐□事故の防止）
- 機能数が多い場合の複数ページ折返し（合計行・備考の改ページ挙動 `wrap`）
- 従来型の参考値・削減率の算出方法（standard/hybrid から計算、ゼロ除算回避）
- `route.ts` の `isValidBody` と新レイアウトで必要なフィールドの整合

## Codex レビュー結果（2026-05-26）

| # | 優先度 | 指摘 | 対応方針 |
|---|---|---|---|
| 1 | P0 | 改ページ制御不足。`wrap={false}` 行は長文 detail でページ高超過時に破綻。合計行・参考行・備考の「同ページ保持」ルール未定義。`fixed` フッターと本文の重なり | テーブル行は `wrap={false}` を外しセル内折返しを許可。合計行＋従来型参考行＋確認事項見出しは `View` でまとめ `minPresenceAhead` で改ページ保護。フッターは `fixed` のまま `paddingBottom` を現状60pt維持し重なり回避 |
| 2 | P1 | Bold サブセットの文字集合運用が未定義。将来ラベル追加で事故る | Bold 対象の静的文字列を**1モジュールに集約**（`PdfBoldGlyphs`）し、そこから文字集合を生成してサブセット作成。`Font.register` は同一 family に `fontWeight:400/700` を登録、style 側で `fontWeight:700` 指定。動的テキストは Bold にしない |
| 3 | P1 | `isValidBody` が `summary`/`discussionAgenda`/`confidence` 欠落を未検証。新デザインで `confidence.rangeLabel` 参照 | これらは任意項目とし、`EstimatePdf` 側で安全フォールバック（`?? ""` / `?? []`）。`confidence` 無しなら精度欄を非表示にする |
| 4 | P1 | 従来型参考値・削減率の算出仕様不足（ゼロ除算・増加時・丸め） | `standard<=0` または `hybrid<=0` の場合は参考行を**非表示**。削減率 = `round((1 - hybrid/standard)*100)`。`hybrid>=standard` なら参考行を出さない。税別前提 |
| 5 | P2 | `totalCost.message` をスキーマから消すとメール文面に影響 | **PDF表示からのみ削除**。型・スキーマ・バックエンド（メール）は変更しない |
| 6 | P2 | 回帰テスト観点不足 | `backend/test_pdf.py` に新項目（confidence あり/なし）・複数ページ・長文 detail・特殊文字のサンプルを追加 |

**結論（Codex）**: レイアウトは `@react-pdf/renderer` v4.3.2 で実現可能。成功の鍵は「改ページ制御」「Bold文字集合の運用自動化」「入力欠落時フォールバック」の3点を設計に織り込むこと。→ いずれも上記方針に反映済み。

## 実装で判明した重大事項（フォント）

実機レンダリング検証で、当初の想定外の不具合を2件発見・修正した。

1. **既存 `NotoSansJP-Regular.ttf` は可変フォントだった**（既定インスタンスが Thin）。`@react-pdf` は可変フォントを埋め込めず、本文が Helvetica にフォールバックして**日本語が全て消えていた**（`¥`→`?`）。→ `instancer` で wght=400 の**静的フォント**に変換して解消。
   - ※ これは今回のリニューアル以前から存在した潜在バグの可能性が高い（本番PDFの日本語表示要確認）。
2. **`@react-pdf` は埋め込みフォントを内部 name で識別**するため、Noto 由来で内部名が同一だった Regular と Bold が**グリフ衝突**し、片方が文字化けした。→ 両フォントを**一意な内部名にリネーム**して解消。

対応として `scripts/build-pdf-fonts.sh` を新設し、可変フォント→静的化→（Boldはサブセット）→一意名リネーム までを再現可能にした。
