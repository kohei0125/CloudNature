## 1. エグゼクティブサマリー

### 戦略の核心

本ツールの最大の武器は **「最短1分で概算 ＋ 稟議に使える開発計画書」** という、BtoB見積もり市場で類を見ない即時性と具体性にある。広告戦略はこの武器を最大限に活かし、**「費用感がわからず動けない」意思決定者の背中を押す** ことに集中する。

### 運用体制の前提

- **運用者は基本的に1名（代表）** — 広告運用の専任担当はいない
- 広告運用は素人からのスタート。まず自分で内容を理解し、小さく始めて学ぶフェーズ
- 外注は検討余地ありだが、初期は自走を優先
- この前提のもと、本ドキュメントの施策は **超ミニマム → 段階的拡大** の順で実行する

### 4つの戦略原則

1. **計測が先、広告は後** — GA4イベント設計・GTM導入・同意バナーを広告配信前に完了させる
2. **Google検索広告1本から始める** — 1人運用では複数チャネル同時は無理。まずは検索広告でCPA実績を取ることに集中
3. **訴求は「速さ × 稟議活用」を主軸に** — 「相場の1/2」等の価格訴求は補助線に留め、信頼リスクを回避
4. **リターゲティングで刈り取る** — BtoB検討期間（数週間〜数ヶ月）を前提とした二段構え（Phase 2以降）

### 現状の最重要課題（広告開始前に対応必須）

| 優先度 | 課題 | 理由 |
|--------|------|------|
| **P0** | 計測基盤の構築（GTM + GA4イベント + 同意バナー） | 計測なしに広告を回すと、最適化の根拠がなく予算が溶ける |
| **P0** | Cookie同意バナーの実装 | 現状GA4が無条件ロード。プライバシーポリシーにCookie利用を明記済みだが同意UIがない |
| **P0** | 社会的証明の追加（最低1件の事例） | 現在ゼロ。実績なしで広告を回してもCVRが死ぬ。自社プロジェクトでもOK。広告開始前に最低1件は必須 |
| **P0** | Step 4（自由記述30字以上）の選択肢化 | 広告で流入を増やしてもStep 4で落ちたらCPAが悪化するだけ。最優先EFO |
| **P0** | フォローメール送付への同意UI追加 | サイト上で「営業メールもお送りしません」と明記しているため、ナーチャリングメールには見積もり完了時の明示的同意が必須 |
| **P1** | 「概算≠正式見積もり」の表現統一 | 利用規約と広告・LPの表現にギャップがあると後工程でクレームになる |
| **P2** | 見積もりアウトプットのサンプル表示 | 「何が出るか」が事前にわかることで開始率が上がる |

## 2. サイト現状分析

### 2.1 強み（広告で活かすべきポイント）

| 強み | 広告への転用 |
|------|-------------|
| 最短1分で概算＋開発計画書まで自動生成 | 検索広告の差別化コピーの核。「待たされない見積もり」 |
| 稟議書として活用可能なPDF出力 | 決裁者・DX担当への訴求。「稟議の一枚目が1分で作れる」 |
| BtoB見積もり系の最大の心理障壁を解消。全広告で強調 |
| チャット形式の選択式UI（13ステップ） | SNS広告で操作画面の動画を見せると「自分でもできそう」感を醸成 |
| AIハイブリッド開発 vs 従来型開発の比較提示 | 「比較できる」こと自体が価値。カルーセル広告で視覚化 |
| SSL暗号化・第三者提供なし・営業メールなしの明示 | 信頼シグナルとして広告LPにも活用 |

### 2.2 課題・改善ポイント

| 課題 | 影響度 | 詳細 |
|------|--------|------|
| **社会的証明がゼロ** | 高 | 実績数・導入企業名・顧客の声が一切ない。BtoB広告のCVRに最も影響 |
| **計測基盤が未整備** | 高 | GA4は入っているがカスタムイベントなし。GTMなし。広告ピクセル（FB/LinkedIn/X）一切なし |
| **Cookie同意バナーなし** | 中（法的リスクは要確認） | GA4が無条件ロード。日本では改正電気通信事業法の対象外（自社サービスの見積もりツール）の可能性があるが、広告計測のConsent Mode連携には必要。法的義務の有無は別途確認 |
| **「相場の1/2」「40%削減」の根拠が薄い** | 中 | 広告審査で不承認リスク。信頼を損ねる可能性。前提条件の明示が必要 |
| **見積もり完了→相談の導線が弱い** | 中 | 完了画面にCTAはあるがデザイン上の強調が不十分。メール内の導線も要強化 |
| **価格訴求とLP/規約の表現にギャップ** | 中 | 利用規約では「AI概算、正式はヒアリング後」と明記。広告で「見積もり」と断言すると齟齬 |

### 2.3 見積もりフローの構造（広告設計上の理解が必要）

```
[13ステップ構成]

Step 1-3:  基本情報（事業形態・業種・規模）
Step 4:    課題・ニーズ（自由記述 30字以上）  ← 離脱リスク高
Step 5-7:  システム要件（対象・種類・開発区分）
Step 8:    機能選択（AIが動的生成）           ← 第1回AI呼び出し
Step 9-11: 条件（納期・デバイス・予算）
Step 12:   補足事項（任意）
Step 13:   連絡先入力 + 同意                  ← 第2回AI呼び出し → PDF生成
```

**広告設計上の注意点：**
- 「最短1分」は選択式のみで進んだ場合。Step 4（自由記述）で時間がかかるユーザーがいる
- Step 8のAI生成機能選択は体験の目玉だが、APIレスポンス待ちが発生する
- Step 13で個人情報を入力するため、ここが最大の離脱ポイント

## 3. ターゲット設計

### 3.0 超ミニマム期の初期ターゲット（Phase 1で集中すべき層）

> **1人運用・月10〜20万円では全方位にアプローチできない。** 初期は以下に絞り、広告コピーの具体性を上げる。

| 項目 | 初期の絞り込み |
|------|--------------|
| **注力業種** | **製造業**（社内ツール・業務効率化のニーズが明確で検索意図が強い）＋ **新潟の全業種**（地域の信頼で商談化率が高い） |
| **注力用途** | 社内業務システム / 業務自動化 / 既存システムの改修・リプレイス |
| **注力KW** | 「システム開発 見積もり」「業務効率化 システム 費用」「新潟 システム開発」等の顕在層KW |

**なぜ製造業か：** 課題が明確（紙・Excel業務のDX化）、検索意図が強い（「製造業 DX」「社内ツール 開発」）、決裁者が見積もり根拠を必要としている。データが溜まったら物流・建設等に横展開する。

### 3.1 プライマリターゲット（全体像 — Phase 2以降で段階的に拡大）

| 属性 | 詳細 |
|------|------|
| **役職** | 情報システム部門マネージャー / DX推進担当 / 経営企画 |
| **企業規模** | 従業員30〜500名、拠点複数、情シスが薄い |
| **業種（最優先）** | 製造、物流、建設、不動産、医療・介護 |
| **業種（次点）** | 教育、士業、小売チェーン本部、B2Bサービス |
| **緊急度** | 3ヶ月以内にPoC or 改修着手を検討中 |
| **共通課題** | 「予算が立たないからプロジェクトが進まない」「見積もりに数週間待てない」「見積もりの妥当性がわからない」 |

### 3.2 セカンダリターゲット

| 属性 | 詳細 |
|------|------|
| **役職** | スタートアップ経営者 / プロダクト責任者 |
| **企業規模** | 1〜30名 |
| **共通課題** | 「MVP開発の費用感を知りたい」「限られた資金で最速で作りたい」 |

### 3.3 ペルソナ別の検索意図と訴求マッピング

| ペルソナ | 検索意図の例 | 刺さる訴求 | 心理的障壁 |
|----------|-------------|-----------|-----------|
| 情シスマネージャー | 「システム開発 見積もり 相場」 | 機能別内訳が出る。稟議資料になる | 見積もりの妥当性への不安。ベンダーロックイン |
| DX推進担当 | 「AI導入 費用 失敗しない」 | AIが要件を分析。従来型との比較ができる | 経営層への説明責任。ROIの根拠 |
| 経営企画 | 「業務効率化 システム 費用」 | 1分で概算。営業電話なし | 見積もり依頼＝営業攻勢を受ける恐怖 |
| スタートアップ経営者 | 「アプリ開発 費用 安く」 | AIハイブリッドで従来比コスト削減 | 予算の制約。失敗への恐怖 |

## 4. 競合分析

### 4.1 国内外の見積もりツール比較

| サービス | 類型 | 入力形式 | 所要時間 | 出力内容 | リード獲得の仕掛け |
|----------|------|---------|---------|---------|-------------------|
| **クラウドネイチャー（当社）** | AI見積 | 選択式13問 | 最短1分 | 費用比較・内訳・体制・スケジュール・PDF | 無料相談予約 |
| deeptier AI見積 | AI見積 | 文章入力 | 約3分 | 費用・体制・スケジュール | 無料CTA |
| Pentagon アプリ見積 | シミュレーター | 選択4問 | 約30秒 | 概算費用（機能別内訳なし） | 要件整理から対応 |
| ドコドア アプリ見積 | シミュレーター | 選択6問 | 約1分 | 概算＋付随費用の%内訳 | 問い合わせ/メール送付 |
| ハコブネ 見積シミュレーター | シミュレーター | 選択式 | 約1分 | 概算（正確な見積はヒアリング） | 「この内容で相談する」 |
| AppCost.AI（海外） | AI見積 | テンプレ/入力 | 約5分 | 機能分解・レポート調整 | 有料プラン＋相談導線 |

### 4.2 当社の競合優位性

**差別化ポイント：**
1. **出力の具体性** — 費用比較（従来 vs AIハイブリッド）・内訳・体制・スケジュールまで出る（競合は総額のみが多い）
2. **稟議資料としてのPDF** — 社内検討にそのまま使える形式は当社のみ
3. **営業電話なし** — 明確に約束している（競合は曖昧）

**競合に劣る点：**
1. **ステップ数（13問）** — Pentagon（4問）やドコドア（6問）に比べて多い。「最短1分」の信頼性にやや影響
2. **実績・社会的証明** — 競合も多くはないが、当社はゼロ
3. **「概算」の前提明示** — ハコブネは「概算の限界」を明示。当社はLP上での説明が不足

### 4.3 SWOT分析

| | 好影響 | 悪影響 |
|---|---|---|
| **内部** | **S:** 即時性（1分）、出力の具体性（PDF・比較・内訳）、営業電話なし、AIハイブリッドの価格優位性 | **W:** 社会的証明ゼロ、価格訴求の根拠不足、計測未整備、13ステップの長さ |
| **外部** | **O:** DX需要の継続拡大、課題記事→見積もりへの導線（既存コンテンツ資産）、AI見積もり市場自体がまだ黎明期 | **T:** 競合の新規AI見積参入（deeptier等）、無料ゆえの低品質リード流入、プライバシー規制強化による計測難化 |

## 6. 広告クリエイティブ・コピー

### 6.1 訴求軸の整理とA/Bテスト設計

広告コピーは **訴求の強さを段階化してA/Bテスト** する。サイト上の強い数値主張（「相場の1/2」「40%削減」）は根拠提示が薄いと不信を招くため、条件付きで使うか強度を落として検証。

| テスト軸 | A案（堅実 — 推奨） | B案（攻め — 要検証） |
|---------|-------------------|-------------------|
| **価値提案** | 最短1分で開発費と体制がわかる | 相場の半分を目指す開発見積もり |
| **不安払拭** | 営業電話なし、概算は無料 | 無料で稟議資料まで作成 |
| **対象明確化** | 社内ツール/業務自動化の費用感 | AIチャットボット/ナレッジ検索の見積もり |
| **行動喚起** | まずは概算をチェック | 失敗しないための見積もりを作成 |
| **訴求の重心** | **信頼性主軸：** 稟議に使える内訳・根拠を提示 | **速さ主軸：** 1分で概算、待たされない |

**初期は A案（堅実）を主軸に配信し、CTR・CVRが安定してからB案をテスト。**

> **訴求優先順位の注意：** BtoBの意思決定者が最も求めているのは「速さ」ではなく「信頼性・正確性・稟議の根拠」。「1分で見積もり」は差別化ポイントだが、それだけを前面に出すと「軽い・安っぽい」印象を与えるリスクがある。初期のテストでは **「稟議に使える内訳＋根拠」を主軸** にし、「速さ」は補助訴求として配置。CTR・CVRのデータで優劣を判断する。

### 6.2 ペルソナ別キャッチコピー

**パターン1: 効率・スピード重視（多忙な決裁者向け）**

> CTA: 「今すぐ見積もりを開始する」

**パターン2: 透明性・信頼重視（慎重な情シス向け）**
> メイン: 「その見積もり、適正ですか？ 開発費の中身を可視化。」
> サブ: 機能別の内訳を提示。従来型開発とAI開発の費用比較で、稟議の根拠が明確になります。
> CTA: 「開発費の内訳をチェックする」

**パターン3: DX推進の共感（経営企画向け）**

> CTA: 「DXの予算相場を調べる」

## 7. LP/CVR最適化

### 7.1 広告開始前に必須の改善（P0/P1）

| # | 改善項目 | 期待効果 | 実装メモ |
|---|---------|---------|---------|
| 1 | **見積もりアウトプットのサンプル表示**（ぼかし入りでも可）をファーストビュー直下に配置 | 「何が出るか」が一瞬でわかり、開始率が上がる | 費用比較・内訳・体制・スケジュールのサンプルを一致させる |
| 2 | **実績の数字を入れる** | BtoB CVRに直結 | 「累計○件の見積もり実績」（少なくてもOK、ゼロよりはるかに良い） |
| 3 | **「概算」の前提をLP上で明示** | クレーム抑制＋誠実さの訴求 | 利用規約の内容を要約:「AI概算であり、正式見積もりは無料相談にて」 |
| 4 | **「相場の1/2」「40%削減」に条件を付記** | 信頼性向上＋広告審査リスク低減 | 「※AIハイブリッド開発の場合。プロジェクトの要件により異なります」 |
| 5 | **お客様の声を1〜2件追加** | 社会的証明 | 短くてもOK。具体的な数字や業種が入っていると効果的 |

### 7.2 見積もりフォーム最適化（EFO）

#### **【最優先】Step 4（自由記述30字以上）の改善**

広告で流入を増やしても、Step 4で離脱されたらCPAが悪化するだけ。**Phase 0〜Phase 1の最優先EFOタスク**として位置づける。

**具体施策：** よくある課題を5つ程度の選択肢として提示し、「その他（自由記述）」を併設する。

例：
- 社内業務の効率化・自動化をしたい
- 既存システムの改修・リプレイスを検討中
- AIを活用した新サービスを開発したい
- お客様向けのアプリ・Webサービスを作りたい
- データ分析・可視化の仕組みを導入したい
- その他（自由記述）

選択肢を選んだ場合は30字制限を不要にし、自由記述は任意化する。これだけでStep 4の完了率が大幅に改善する見込み。

#### その他のEFO施策

| 改善テーマ | 具体施策 | 期待効果 | 優先度 |
|-----------|---------|---------|--------|
| **進捗の明示** | 「残り○問」「全体の○%完了」を常時表示 | 完遂意欲の維持。離脱率低下 | P1 |
| **Step 13（連絡先入力）の離脱対策** | 信頼シグナル（SSL・第三者提供なし・営業メールなし）を入力欄の直上に再掲 | 最終ステップの離脱防止 | P1 |
| **概算レンジの先出し（段階的開示）** | Step 12完了時点で概算の総額レンジ（例：「300〜500万円」）を表示し、詳細（PDF・内訳・体制）は連絡先入力後に提供する設計 | Step 13の入力動機を強化。「詳細を見るために連絡先を入力する」という合理的な理由が生まれ、離脱率が下がる | P1 |
| **入力完了後の即時フィードバック** | 見積もり結果の一部（総額レンジ等）を完了画面で即時表示 | 「メール待ち」のフラストレーション軽減 | P2 |

### 7.3 見積もり完了画面の最適化

| 改善項目 | 施策 |
|---------|------|
| **主CTAの視覚的強調** | 「無料相談を予約する」ボタンをより目立つ位置・サイズに |
| **結果のメール送信訴求** | 「この見積もり結果を社内検討用にメールで保存」→ メール取得の根拠 |
| **類似事例の表示** | 「この予算帯での開発事例」を完了画面に連動表示 |
| **次のアクションの明確化** | 「概算→無料相談（30分）→正式見積もり」のステップを図示 |

### 7.4 モバイル最適化

BtoB商材であっても、SNS広告経由のアクセスはモバイルが主流。

- タップしやすい大きなボタン（最低44px）

### 8.1 現状の課題

| 項目 | 現状 | あるべき姿 |
|------|------|-----------|
| GA4 | ✅ 導入済み（G-1CF4H5GXSM） | カスタムイベント追加が必要 |

### 8.2 推奨タグ構成

```
サイト / 見積もりツール
    │
    ├── GTM Web Container
    │     ├── GA4 （既存ID移行）
    │     ├── Google広告 コンバージョンタグ
    │     ├── Google広告 拡張コンバージョン（リード）
    │     ├── Meta Pixel
    │     ├── LinkedIn Insight Tag
    │     ├── X (Twitter) Pixel
    │     └── Consent Mode 連携
    │
    └── CRM / スプレッドシート / Notion
          └── オフライン成果（商談・受注）
                └── Google広告 拡張コンバージョン（リード）にフィードバック
```

### 8.3 推奨イベント設計

| ファネル段階 | イベント名 | 主要パラメータ | キーイベント化 |
|------------|-----------|--------------|--------------|
| 見積もり開始 | `estimate_start` | 流入元、utm_source/medium | しない（質が落ちる） |
| 各ステップ到達 | `estimate_step` | step_number (1-13) | しない |
| Step 4完了（自由記述） | `estimate_step_freetext` | text_length | しない（離脱分析用） |
| Step 8完了（AI機能選択） | `estimate_step_ai_features` | selected_count | しない |
| **見積もり完了** | **`generate_lead`** | 規模, カテゴリ, 想定費用レンジ | **する** |
| PDF出力/メール送付 | `estimate_pdf_sent` | — | しない |
| **無料相談予約/問い合わせ** | **`contact_submit`** | フォーム種別 | **する** |

### 8.4 広告コンバージョンの設定

- **主KPI（入札最適化の対象）:** `contact_submit`（無料相談・問い合わせ）
- **準KPI（学習補助）:** `generate_lead`（見積もり完了）
- **注意:** `estimate_start` を主KPIにすると質の低いリードに最適化されるため、必ず上位CVと併用
- **※超ミニマム期間中の例外：** 月10〜20万円の初期段階では `contact_submit` の月間CV数が自動入札の学習に必要な目安（月30件程度）に届かない可能性が高い。Phase 1では `generate_lead`（見積もり完了）を入札最適化の対象とし、月間CV数が安定してから `contact_submit` に切り替える

### 8.5 拡張コンバージョン（リード）の設計

BtoBはオンラインで完結しない。フォームで取得したメールアドレスをハッシュ化してGoogle広告に送信し、オフライン成果（商談・受注）と紐付ける。

**実装フロー：**
1. `contact_submit` 時にメールアドレスをSHA-256ハッシュ化
2. Google広告の拡張コンバージョン（リード）タグで送信
3. 商談・受注が確定したら、CRM/スプレッドシートからオフラインコンバージョンをアップロード
4. Google広告の自動入札がオフライン成果まで含めて最適化

### 8.6 Cookie同意設計

**現状のリスク:** GA4が無条件ロード。プライバシーポリシーにCookie利用を明記しているが、ユーザーの同意取得UIがない。

**推奨設計：**
1. Cookie同意バナーを実装（初回訪問時に表示）
2. Google Consent Mode v2 と連携
3. 同意状態に応じてタグの挙動を制御：
   - 同意あり → 全タグ発火
   - 同意なし → Consent Modeの匿名計測（モデリング）
4. 同意バナーのUXは「承認」「拒否」「設定」の3択が標準

## 9. SEO・コンテンツ戦略

### 9.1 基本方針

見積もりLPは単一ページ型であり、SEO単体での集客力は限定的。既存の「AI導入のヒント・実践ガイド」記事群を入口にし、見積もりツールへ送客する **二段構え** が現実的。

```
[課題記事: SEOで集客]  →  [見積もりツール: CVで獲得]  →  [無料相談: 商談化]
```

### 9.2 コンテンツ拡充の方向性

| コンテンツカテゴリ | 狙うキーワード例 | 記事→見積もりへの接続 |
|------------------|-----------------|---------------------|
| **費用相場系** | 「AI開発 費用 相場 2026」「チャットボット 開発 いくら」 | 記事の結論直後に「あなたのケースの概算はこちら→見積もりツール」 |

## 11. 予算シナリオと KPI

### 11.1 許容CPA の算出（自社の受注単価から逆算）

広告のCPA目標は市場相場ではなく、**自社の受注単価と粗利から逆算** すべき。

```
許容CPA = 平均受注単価 × 粗利率 × 見積もり→商談化率 × 商談→受注率 × 目標ROAS⁻¹

例: 受注単価300万円 × 粗利率40% × 商談化率20% × 受注率25% × (ROAS 300%)⁻¹
  = 3,000,000 × 0.4 × 0.2 × 0.25 × 0.33
  = 約20,000円（見積もり完了1件あたりの許容CPA）
```

**実際の数値は受注実績に基づいて調整すること。**

### 11.2 予算シナリオ（4段階）

| シナリオ | 月額広告費 | 配分 | 目的 | 想定月間リード | 運用負荷 |
|---------|----------|------|------|--------------|---------|
| **超ミニマム（推奨スタート）** | 10〜20万円 | Google検索 100% | 検索広告1本でCPA実績を取る。KWと広告文の勝ちパターンを発見する | 3〜8件 | 1人で可能 |
| **ミニマム検証** | 20〜40万円 | 検索80% / リタゲ20% | 勝ちパターンを拡大＋リタゲ開始 | 10〜20件 | 1人で可能 |
| **標準** | 60〜120万円 | 検索55% / SNS20% / リタゲ15% / 動画10% | 複数チャネルで理解度を上げ、相談転換を狙う | 30〜60件 | 外注検討 |
| **攻め** | 150〜300万円 | 検索45% / SNS20% / リタゲ15% / 動画15% / 提携5% | フルファネルで指名検索増と商談量を取りに行く | 80〜150件 | 外注推奨 |

**推奨：** まずは **超ミニマム（月10〜20万円）** でGoogle検索広告1本に絞り、2〜3ヶ月かけてCPA・CVRの実績値を取得する。1人運用ではここに集中するのが現実的。実績が見えてからミニマム検証→標準へ段階的に拡大。

#### 超ミニマムシナリオの詳細

- **チャネル：** Google検索広告のみ
- **KW：** 顕在層（「システム開発 見積もり」等）＋ 新潟ローカルKW に絞る
- **広告文：** 堅実A案（§6.1参照）を3〜5本でテスト
- **日予算：** 3,000〜6,000円/日（月9〜18万円）
- **学習期間：** 最低2週間はデータ蓄積に充てる。初動で判断しない
- **拡大判断基準：** CPA（見積もり完了）が20,000円以下で安定したら予算増額を検討

## 12. リスク管理

### 12.1 広告審査リスク

| リスク | 対象 | 対策 |
|-------|------|------|
| **「相場の1/2」「40%削減」の根拠不足** | Google / Meta 広告審査 | 広告文では「コスト削減を実現※」＋LP上に「※AIハイブリッド開発の場合。要件により異なります」を明記 |
| **「最短1分」の誇大表現リスク** | 全チャネル | 「※選択式の場合の目安。自由記述の内容により変動します」を注記 |
| **比較広告規制** | Google / Meta | 「SIerより安い」等の直接比較は避け、「従来型開発との比較ができる」に留める |

### 12.2 低品質リードの流入リスク

| リスク | 原因 | 対策 |
|-------|------|------|
| 冷やかし・競合調査の流入 | 無料ツールの宿命 | コンバージョンの最適化対象を `contact_submit` に設定し、`estimate_start` で最適化しない |
| 学生・個人の流入 | 興味本位のアクセス | 除外KW設定（「無料ツール」「学習」等）＋ LP上に法人向けであることを明記 |
| BtoB検討期間の長さ | 見積もり完了→商談まで数週間〜数ヶ月 | リターゲティング＋メールナーチャリングで長期フォロー |

### 12.3 プライバシー・法的リスク

| リスク | 対策 |
|-------|------|
| Cookie同意なしでのGA4運用 | 日本では改正電気通信事業法の対象外の可能性あり（自社サービスの見積もりツール）。ただし広告計測のConsent Mode連携には同意バナーが必要。法的義務の有無は別途確認し、少なくとも広告配信チャネルの要件に合わせて対応 |
| 広告計測のプライバシー規制強化 | 拡張コンバージョン（リード）でファーストパーティデータ活用。サーバーサイドGTMの検討 |
| 見積もりデータの取扱い | プライバシーポリシーと利用規約の表現をLP上で要約表示 |

### 12.4 「概算≠正式見積もり」の認識ギャップ

**リスク:** 広告やLPで「見積もり」と表現し、ユーザーが正式見積もりだと誤解→無料相談時に金額が異なりクレーム。

**対策:**
- 広告文で「概算」「シミュレーション」を一貫して使用
- LP上に「本見積もりはAIによる概算です。正式なお見積もりは無料相談にてご案内します」を見積もり開始前に表示
- 結果画面の免責文言を視認しやすい位置に配置
- メールにも同様の文言を記載
