## あなたの役割

株式会社クラウドネイチャーのコーポレートサイト cloudnature.jp を改修する。**コピー（文言）の修正と技術的な修正のみ**を行う。

## スコープの絶対条件

**サイトの構成は変更しない。** セクションの追加・削除・並び替え、サービスの掲載順の変更、ナビゲーションやフッターの構造変更は一切行わない。変更してよいのはテキスト・文言・注記・リンクテキストと、P2で指定する技術的修正のみ。

## 前提知識（ブランド方針）

- 提供サービス：**1. システム開発 2. AIエージェント開発 3. 法人向けAI導入支援**。メイン事業はシステム開発・AIエージェント開発。個人向けスクール「新潟AIアカデミー」はサブ扱いで、サイト上の現在の扱い（リンク導線）を維持する。サービスは変更しなくて良い。
- ターゲット：年商5,000万〜5億規模の中小企業。システム開発費に300万円以上出せる企業
- 価格方針：安売りしない。**価格の安さを訴求軸にしない**
- 販売プロセス：無料相談 → 提案・デモ → **精度検証（PoC・有償）** → 本開発 → 運用・定着支援
- 強み（コピー執筆時のトーン基準）：AI前提設計によるスピードと柔軟性／試験導入で終わらせず定着まで一貫支援／戦略設計から開発まで一貫した体制／経営課題を技術要件へ翻訳する要件定義力
- 表現ガードレール：AIについて「完全自動」「完全に学習」等の断定表現を使わない。「確認を支援」「候補を抽出」「〜を支援」等の表現を使う。成果数値は根拠のある事例ベースでのみ記載

## 禁止事項（絶対厳守）

1. 「フリーランス」という語をサイトのどこにも書かない
2. 会社の設立年月日を記載しない。あわせて「創業以来」「長年の実績」など**社歴を示唆する表現も一切書かない**（代表個人の経験年数への言及は可）
3. コスト削減率・価格の安さを訴求コピーにしない
4. 事実確認できない数値・実績・顧客の声を新規に創作しない
5. セクション構成・掲載順・ナビ構造を変更しない

## 修正タスク（P0 → P1 → P2 の順に実施）

### P0-1｜代表メッセージの経歴段落を差し替え（/company）

現在の経歴に関する記述（「東京・大阪と、数十万人が毎日使うサービス〜」を含む段落）を、以下の文に**一字一句このまま**差し替える。経歴以外の段落は保持する。

> 新潟のIT企業からキャリアをスタートし、その後は独立して東京・大阪の企業を中心に、数十万人が毎日使うサービスの開発や大手企業のシステム構築に携わってきました。約8年間、開発の現場に立ち続ける中で確信したのは、AIが企業の経営を根本から変える力を持つということです。AIやシステムの力は、まだ全ての企業に届いているわけではありません。もっとできることがあるはず。その想いを胸に、クラウドネイチャーを立ち上げました。

### P0-2｜価格訴求コピーの撤去・言い換え（全ページ横断）

サイト全体を「40%」「コスト削減」「相場」で全文検索し、**開発費の安さを訴求するコピーをすべて書き換える**。言い換えの方針：

- 訴求軸は「スピード」と「スコープ」に転換する
- 言い換え例：「AIを前提とした開発体制により、同じ予算でより広い範囲の業務を、より短い期間で仕組み化できます」
- 削減率の数字を残す場合は、顧客側の**業務工数削減**（開発費ではない）に限定し、事例根拠がある場合のみ

### P0-3｜「導入の流れ」の文言修正（/services）

既存の「導入の流れ」セクション内で、ステップの文言を以下に差し替える（セクション自体の位置・デザインは変更しない）：

1. **無料相談・業務ヒアリング** ── 「お困りごと」ではなく、業務の事実を一緒に棚卸しします
2. **ご提案・デモ** ── 課題に合わせた解決アプローチと概算をご提示します
3. **精度検証（PoC・有償）** ── 実際の業務データで小さく検証し、効果を数値で確認します（検証費用は本開発ご契約時に全額充当）
4. **本開発** ── 検証結果に基づき、本番システムを開発します
5. **運用・定着支援** ── 導入後の改善・定着まで伴走します（保守・月次改善）

※金額は記載しない。「有償」であることと「本開発契約時に充当」のみ明記。
※現在のステップ数と異なる場合は、既存のステップ表示コンポーネントの範囲内で数を合わせて調整する（新規セクションは作らない）。

### P1-4｜「平均80%の工数削減」の修正（/cases）

「平均」を「最大」に変更し、注記を追加する：
「※削減率は特定事例の実績値です。効果は業務内容・運用状況により異なります。」

### P1-5｜導入事例の帰属表記の確認（/cases）

「AI学習管理システム（研修サービス企業）」の事例について、これが自社サービス（新潟AIアカデミー）向けの開発である場合は「自社サービスとして開発・運用」と明記に修正する。外部顧客の実案件であれば変更不要。**コードやコンテンツから判断できない場合は変更せず、確認事項として報告する。**

### P2-6｜CTA文言の再フレーム（トップページ）

ボタンの構成・配置・数は変更しない。AI見積もり系CTAの文言のみ、価格提示ツールではなく技術力のデモとして伝わる表現に変更する。
例：「自社開発のAI見積もりを体験（最短1分）」

### P2-7｜技術スタック表記の統一

サイト内の技術スタック表記を以下に統一する：
Next.js / TypeScript / React / Python / AWS / Google Cloud / Claude / OpenAI

PHP の表記は、実際にPHP案件を提供している場合を除き削除する。判断できない場合は削除せず確認事項として報告する。

### P2-8｜技術チェック

- H1タグがページ内で重複していないか確認（レスポンシブ用の複製がある場合、SEO上H1が1つになるよう修正）
- 変更後の内部リンク切れがないか全ページ確認
- 既存の title / meta description は原則変更しない（SEO維持）。ただし今回のコピー変更と明確に矛盾する箇所のみ最小限修正

## 作業手順（必ずこの順で進める）

1. **Blind Spot Pass**：実装に入る前に、対象ページとコンポーネント構成、コピーの管理場所（ハードコード／CMS／定数ファイル／多言語ファイル等）を把握し、影響範囲と盲点を列挙する
2. **Implementation Plan**：修正一覧・対象ファイル・変更方針を提示する。判断が必要な点（P1-5の事例帰属、P2-7のPHP）はこの段階で質問する
3. **実装**：計画の承認後に着手する。1タスク＝1コミット相当の粒度で進める
4. **Verification**：ビルド・lint・全ページの表示確認・リンクチェックを実施する。確認できなかった項目は確認済みのように書かない
5. **報告**：変更内容／Before・After／確認方法／未確認事項・残リスクを分けて報告する
